1. Company Overview

Bartez Tecnología is a B2B wholesale distributor of technology products — including laptops, desktop computers, PC components, networking equipment, storage, and peripherals — serving IT resellers, system integrators, and corporate buyers in Argentina. The company operates a proprietary B2B portal at bartez.com.ar where registered business customers can browse catalog, request quotes, place bulk orders, and manage their accounts.

2. Purpose of the Integration

We use Google Ads exclusively to acquire new B2B resellers and corporate buyers for our own business. We do not manage advertising for third parties, and the tool is not offered as a product to external clients. The integration automates two internal workflows that are currently handled manually:

  1. Campaign creation: our marketing team generates campaign drafts targeted to specific B2B segments (e.g. IT resellers, corporate procurement), reviews them internally, and pushes approved drafts to Google Ads via the API. This eliminates repetitive manual entry of ad groups, keywords, headlines and descriptions in the Google Ads web UI.
  2. Performance reporting: a daily scheduled job pulls impressions, clicks, cost and conversions from Google Ads into our internal analytics dashboard, so we can correlate ad spend with downstream B2B events (registration, first order, repeat purchase) tracked in our own database.

3. System Architecture

┌──────────────────────┐ ┌─────────────────────────────┐ │ Admin UI (React) │ │ Supabase Edge Functions │ │ bartez.com.ar/admin │ ──────▶ │ (Deno runtime) │ │ Marketing tab │ │ │ └──────────────────────┘ │ • launch-campaign │ ▲ │ • google-ads-sync (cron) │ │ └─────────────┬───────────────┘ │ │ │ ▼ │ ┌─────────────────────────────┐ │ │ Google Ads API (REST) │ │ │ googleads.googleapis.com │ │ └─────────────┬───────────────┘ │ │ ▼ ▼ ┌──────────────────────┐ ┌─────────────────────────────┐ │ Supabase Postgres │ ◀────── │ ad_performance_snapshots │ │ campaign_drafts │ │ ad_campaigns │ └──────────────────────┘ └─────────────────────────────┘

Components

  • Admin UI — React SPA accessible only to authenticated internal admin users. Marketing team generates drafts, reviews them, and triggers launches.
  • launch-campaign (Edge Function) — Receives an approved draft ID, authenticates the caller, calls Google Ads API to create budget → campaign → ad groups → ads → keywords, and stores the returned campaign ID back in the draft record.
  • google-ads-sync (Edge Function, cron daily) — Runs once per day (06:00 UTC), reads the previous day's performance via googleAds:searchStream, and upserts the rows into ad_performance_snapshots.
  • Supabase Postgres — Stores draft structures, campaign metadata, and performance snapshots for our internal analytics.

5. Authentication & Authorization

  • OAuth 2.0: Server-side OAuth with refresh token stored as an environment secret in Supabase Edge Functions. Token was issued for an internal Google account with admin access to the MCC.
  • Manager account operation: The refresh token belongs to the manager account 897-050-2244 (Bartez Tecnología). Mutations and reads are performed against the client account 969-460-7792 using the login-customer-id header set to the MCC ID.
  • Internal auth: Every Edge Function call validates the Supabase session and verifies that the caller has the admin role in the profiles table before invoking Google Ads.
  • No end-user access to API: End users of bartez.com.ar (B2B customers) never interact with the Google Ads API, directly or indirectly. Only internal marketing/admin staff can trigger API calls.

6. Safety and Cost Controls

  • Every campaign created via the API is initialized with status: "PAUSED". Activation is always done manually by a human inside the Google Ads web UI after visual review.
  • Daily budgets are capped per campaign (configured by the admin in the draft before approval).
  • All draft structures go through a mandatory human approval step before the launch-campaign function becomes callable for that draft.
  • Headlines are truncated to the Google Ads limit (30 characters) and descriptions to 90 characters client-side before the API call, to avoid validation rejections.
  • Negative keywords are applied at campaign level to reduce wasted spend on unrelated searches.

8. Data Handling

  • No personal data of Google Ads end-users (ad viewers/clickers) is stored — only aggregated metrics (impressions, clicks, cost, conversions) at campaign level.
  • Campaign drafts, structures, and snapshots are stored in Supabase Postgres with row-level security restricting access to admin users.
  • OAuth credentials (developer token, client secret, refresh token) are stored as Supabase Edge Function Secrets — never committed to source control and never exposed to the browser.

9. Privacy & Compliance

  • The B2B portal bartez.com.ar has a published privacy policy describing data handling.
  • The tool complies with Google Ads API Terms of Service and Required Minimum Functionality (RMF) requirements for internal-use tools: UI provides campaign creation, status visibility, performance reporting, and the ability to pause/review campaigns.
  • No reselling of Google Ads data, no white-labeling, no multi-tenant use.

10. Summary

This is a single-company internal tool used exclusively to manage Bartez Tecnología's own Google Ads account. It replaces repetitive manual work in the Google Ads UI with an approval-based automation pipeline. All campaigns created via the API require human review before activation, and all performance data flows into our internal analytics dashboard for attribution. We request Basic Access to allow the integration to operate against our production Google Ads account (969-460-7792), managed through our MCC (897-050-2244).
